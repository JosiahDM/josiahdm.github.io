Java Basics: import

Java has a huge amount of built in tools that we can leverage to build software. This is a huge benefit because we don't have to build these tools ourselves from the ground up, which can save us a lot of time and allow us to focus on MAKING things that work for people, and not necessarily needing to know how it's working at every possible layer 100% of the time.

Of course I do think we should strive to learn as deeply as we can on many topics, we don't need 100% understanding to make useful things. If we waited until that point we'd never make anything! To quote a great blog that I love:

"That is why big companies hire young, enthusiastic coders. Not because they're as clueless as my madman, but because they can self-motivate beyond the barrier of full understanding and into imperfect and brute force solutions. I'd want to stop and rework the abstractions I'm using and break things into lots of smaller, reliable, understandable pieces. My code might be more bullet-proof in the end, but I still have a level of admiration for people who can bang out complex apps before they become jaded enough to realize it's not that easy."

So to come back to the topic of the article...using import (and the many built in Java packages) gives us a ton of cool things that can make our lives easier and more interesting.

So what does import do?

When you write import java.util.Scanner; you can think of it as "importing" all of the functionality of the java.util.Scanner class into your code, and you can treat Scanner as if it was already "inside" your code.

It doesn't actually import the code, however. What it does is tells the compiler where to look for "Scanner" when you decide to create a new Scanner object.

Without import you'd have to write java.util.Scanner keyboard = new java.util.Scanner(System.in); This works but it's much nicer to be able to just write Scanner keyboard = new Scanner(System.in);. Using import tells the compiler "when I write Scanner, I mean java.util.Scanner, k thanks, you know what to do."

What's this java.util business?

In kind of the same way we have directories in our PCs, Java has "directories" called packages. Our different PC directories are like /etc or /usr (or if you use windows: C:\Program Files, C:\System...etc). But we can go further down into the hierarchy with /usr/bin/... packages work in much the same way.

Top level packages include java, javax, and org. Within some packages we can drill down further. For example java.util to java.util.concurrent. Much like directories contain files and other directories, packages contain Classes, Interfaces, and other packages and Java structures.

Packages help us keep this huge amount of tools organized and logically separated. Also, it keeps code in separate "namespaces" which means we can avoid conflicts in naming things. So if for whatever reason I want to make a class named "Security", it won't conflict with the already made java.security.Security class, since it is in a different namespace. But if I tried to import java.security.Security at that point, it would conflict with my custom Security class.

My arbitrary example might suck, but the main point is that it keeps things organized and separated when dealing with a huge code base that might otherwise be a jumble of chaos, confusion, and sadness.
